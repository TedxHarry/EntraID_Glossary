# Security Alert: When Entra ID Tells You Something Is Wrong

**Part of Entra ID Glossary Series: Glossary#12.4 - Security Alert**

---

A Monday morning. Identity team arrives. Overnight, Entra ID generated 14 security alerts. Eleven were "unfamiliar sign-in properties" for users who signed in from their home offices over the weekend, which the system hadn't seen before. Two were "atypical travel" for a VP who had flown cross-country Friday afternoon. One was "leaked credentials" for a service account whose password appeared in a data breach database.

The eleven were false positives from weekend remote work patterns. The two were explainable by a known travel event. The one was a real finding that needed immediate action.

This is what security alert triage looks like in practice: most are noise, one is signal. The skill is knowing which is which quickly.

## ⚠️ What Security Alerts Are

Entra ID security alerts are automated detections generated by Microsoft Entra ID Protection and other identity security services when suspicious activity is detected. They represent the output of risk detection: the machine learning models, threat intelligence feeds, and behavioral analysis that runs continuously across authentication events.

Each alert represents a specific detection type with a specific risk signal. Not every alert is a confirmed compromise. Alerts indicate potential risk, not certain harm. Investigation determines the actual impact.

## 🔍 Alert Detection Types

Entra ID Protection generates alerts across several detection categories:

**Credential-based detections** 🔑:
- **Leaked credentials**: Microsoft's threat intelligence has found credentials matching this account in public breach databases, dark web data dumps, or paste sites. High severity. Requires immediate password reset and investigation of what was accessed.
- **Password spray**: Multiple accounts in the tenant being tested with the same common password from one or a few IP addresses. The attacker is trying to find accounts with weak passwords without triggering individual account lockouts.

**Location-based detections** 📍:
- **Impossible travel**: Sign-ins from two locations that are geographically impossible to reach sequentially in the elapsed time. Most common false positive source for remote workers and VPN users.
- **Unfamiliar sign-in properties**: A sign-in with properties (location, device, browser, ASN) that differ significantly from the user's established pattern. Context-dependent false positive rate.
- **Anonymous IP**: Sign-in from a Tor exit node or known anonymous proxy. Legitimate privacy-conscious users use these, as do attackers.
- **Malicious IP**: Sign-in from an IP with confirmed malicious activity in Microsoft's threat intelligence.

**Behavioral detections** 🎭:
- **Token theft**: Signals suggesting a valid token is being used from an IP different from where it was acquired. Increasingly relevant with adversary-in-the-middle phishing attacks that steal tokens.
- **Suspicious browser**: Browser or user agent string associated with malicious activity.
- **Mass access to sensitive files**: Bulk file access suggesting data exfiltration (fed from Microsoft Defender for Cloud Apps integration).

## 🔄 Alert Lifecycle

Alerts flow through a lifecycle from detection to resolution:

**Detected** 🔍: Entra ID generates the alert. Appears in the ID Protection risk detections blade and Microsoft Defender XDR.

**Reviewed** 📋: A security analyst examines the alert. Reviews the sign-in logs associated with the detection. Checks whether the behavior is explained by a known event (travel, new device, VPN usage).

**Confirmed compromised** ⚠️: The analyst determines the detection represents a real threat. The user's account is flagged as compromised. Remediation begins: force password reset, revoke sessions, investigate what was accessed.

**Dismissed** ✅: The analyst determines the detection is a false positive. Documenting the reason (known VPN, travel, new device) improves future triage context.

**Remediated** 🔒: For confirmed compromises, the security incident is resolved. Account secured, access reviewed, impact assessed.

## 📊 Where Alerts Surface

**Entra ID Protection blade** 🔵: The primary interface for ID Protection risk detections. Shows risky users and risky sign-ins, with the ability to investigate and remediate.

**Microsoft Defender XDR** 🛡️: Identity alerts from Entra ID appear alongside endpoint, email, and application alerts in the unified security operations center experience.

**Microsoft Sentinel** 📈: When Entra ID diagnostic settings export data to Log Analytics, Sentinel analytics rules can generate incidents from sign-in and risk data. Custom alert logic beyond what ID Protection provides natively.

**Email notifications** 📧: Configurable in ID Protection settings to send email alerts when high-risk detections occur. Useful for teams not monitoring dashboards continuously.

## ⚙️ Automated Response with Conditional Access

The most operationally efficient approach to security alerts isn't manual triage for every detection; it's automating the response through Conditional Access risk policies:

- User risk policy: When user risk is High, require password reset to regain access.
- Sign-in risk policy: When sign-in risk is Medium or High, require MFA to complete authentication.

These policies block or remediate risk detections in real time without analyst involvement for the most common scenarios, reserving human triage for alerts that require investigation beyond automated remediation.
