# Authentication Method
*Not All Proof Is Created Equal*

> **Difficulty:** 🟡 Intermediate

📚 **Part of Entra ID Glossary Series: Glossary#3.3 - Authentication Method**

---

## 🎯 TL;DR

- Authentication methods are the mechanisms users prove identity: password, TOTP, phone call, FIDO2 key, etc.
- The Authentication Methods Policy controls which methods are available and for which users in your tenant
- Stronger methods (FIDO2, WHfB) are phishing-resistant; TOTP and SMS are not


In 2022, a mid-sized organization I worked with had a credential breach. An attacker bought a list of username and password combinations from a dark web breach dump, ran them against the company's login page, and found 47 accounts where the password matched. They'd implemented MFA, though. The attacker was stopped... for about four hours.

Then they started calling users on the phone. "Hi, I'm from IT support. We're doing an urgent security check and need you to approve the MFA request you're about to receive." Twelve users approved it. The attacker was in.

The organization had authentication. They had MFA. What they didn't have was the *right* authentication method for the threat they faced.

## 📊 The Spectrum of Authentication Methods

Authentication methods in Entra ID range from highly phishable to completely phishing-resistant. Where a method falls on that spectrum matters more than whether you have MFA at all.

**🔴 Password only**
The baseline. A shared secret that can be guessed, phished, stuffed from breach lists, or stolen by malware. Passwords alone are not sufficient for any meaningful security in 2026.

**🟠 SMS one-time passcode**
A code sent to a registered phone number. Better than password alone, but SMS can be intercepted via SIM swapping. And as the story above shows, users can be socially engineered into reading the code out loud to an attacker.

**🟡 Voice call**
User receives a call and presses a key to approve. Same SIM swapping risk as SMS. Same social engineering susceptibility.

**🟡 TOTP (time-based one-time codes)**
The 6-digit codes generated by authenticator apps. Better than SMS because they're generated locally, not transmitted. But they're still phishable: a real-time phishing site can relay the code to the actual service before it expires.

**🟢 Microsoft Authenticator push notification (with number matching)**
A notification sent to the registered device. The user sees a number on screen and must match it in the app before approving. This kills the "approve everything" fatigue attack and makes blind approvals much harder. Not phishing-resistant, but significantly more resistant than SMS or plain push.

**🟢 Certificate-based authentication**
Smart cards or certificates on devices. Strong, requires physical possession. Common in government and regulated industries.

**✅ FIDO2 security keys**
Hardware keys (YubiKey, Feitian, etc.) that are cryptographically bound to the site they're registered with. Completely phishing-resistant by design. If someone creates a fake login page, the key won't authenticate to it.

**✅ Windows Hello for Business**
Biometric or PIN authentication backed by device-bound cryptographic keys. Phishing-resistant. The credential never leaves the device.

## 🔧 Authentication Methods Policy in Entra ID

Administrators control which authentication methods are available to users through the Authentication Methods Policy in Entra ID. This is where you enable or disable Microsoft Authenticator, FIDO2, SMS, voice calls, certificate-based auth, and others.

⚠️ One important distinction: there are two places where authentication methods get configured and they overlap in confusing ways. The **Authentication Methods Policy** (under Protection > Authentication Methods) is the modern, converged policy. The older **MFA settings page** is a legacy location that's been gradually deprecated. Microsoft is migrating everything to the Authentication Methods Policy. If you're seeing inconsistent behavior, check whether settings conflict between the two locations.

## 🎯 Authentication Strength in Conditional Access

Starting in late 2022, Conditional Access gained a "Require authentication strength" grant control. Instead of just saying "require MFA," you can specify *which* MFA methods satisfy the requirement.

Built-in authentication strength levels:

- **Multifactor authentication:** any two-factor combination (including SMS)
- **Passwordless MFA:** Windows Hello for Business, FIDO2, or Authenticator passwordless
- **Phishing-resistant MFA:** Windows Hello for Business or FIDO2 only

You can also define custom authentication strengths for specific scenarios.

This matters enormously. "Require MFA" for administrators that allows SMS as a valid method means an admin account can be compromised via SIM swap or a phone call. Scoping admin Conditional Access policies to require phishing-resistant MFA closes that gap.

## 💡 The Practical Takeaway

Requiring MFA is the right baseline. But don't stop there. Look at *which* methods satisfy your MFA requirements for high-value scenarios.

For administrators and privileged roles: phishing-resistant methods only. FIDO2 keys or Windows Hello for Business.
For standard users: Microsoft Authenticator with number matching at minimum. SMS only where no other option exists, and with a plan to migrate off it.

The attacker who called twelve users and got them to approve push notifications would have had zero success against phishing-resistant MFA. The credential breach becomes a non-event.
