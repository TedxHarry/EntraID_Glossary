# Authentication Methods (Security Focus)
*Choosing the Right Mix for Your Security Posture*

**Part of Entra ID Glossary Series: Glossary#7.26 - Authentication Methods (Security Focus)**

---

A security team completed an MFA deployment. 100% of users had MFA registered. They reported success.

Then the breach happened. A targeted phishing campaign got a senior manager's TOTP seed compromised. An attacker who had the password and the TOTP secret could generate valid MFA codes without the user's phone. The TOTP seed had been extracted from a backup of the user's old phone that was stored unencrypted in a cloud storage service.

The organization had the right MFA policy. They had the wrong MFA methods enabled.

## 🔍 Authentication Methods as a Security Decision

Authentication methods in Entra ID aren't just a user experience setting. The methods you enable, require, or allow determine the attack surface of your identity platform. Every method has specific vulnerabilities. Every method has specific strengths. Security-focused authentication method management means choosing methods whose vulnerabilities are acceptable for the accounts and resources they protect.

Entra ID's Authentication Methods settings (under Security > Authentication Methods) let you enable or disable specific methods tenant-wide and configure them. The decisions you make here determine what's available for MFA, SSPR, and Conditional Access.

## 📱 The Method Landscape

**Microsoft Authenticator** 🔔: The recommended primary method for most organizations. Push notifications require the user to approve a request on their registered phone. Number matching (now required by default) adds a verification step: the user sees a number on the sign-in screen and must enter it in the app, preventing blind approval of attacker-initiated prompts.

Passwordless phone sign-in via Authenticator is available: users approve sign-ins with biometric or PIN without ever entering a password. Stronger than push notification + password.

**TOTP authenticator apps** 🔢: Time-based one-time password apps (Microsoft Authenticator can also generate these, as can Google Authenticator and others). 6-digit codes that change every 30 seconds. The TOTP shared secret is stored in the app. If the seed is exposed (backup compromise, screen capture, app extraction), an attacker can generate valid codes without the device.

**SMS and voice** 📲: One-time codes delivered to a registered phone number. SMS is vulnerable to SIM swapping (an attacker convinces the carrier to transfer the number) and to interception. Voice is similarly vulnerable. Both should be avoided for high-value accounts. They're acceptable as backup methods for lower-sensitivity scenarios.

**FIDO2 security keys** 🔑: Hardware security keys using the FIDO2/WebAuthn standard. Phishing-resistant. The private key never leaves the hardware. Origin-binding means credentials won't authenticate against a phishing domain. Requires users to carry a physical key. Loss of the key requires re-enrollment.

**Windows Hello for Business** 🪟: Biometric or PIN-based authentication on Windows devices. The credential is device-bound using the TPM chip. PIN isn't a password: it's a local secret that unlocks a device-bound cryptographic key that's never transmitted. Phishing-resistant. Requires configured deployment to work correctly.

**Certificate-based authentication** 📜: Smart card or software certificate authentication. Common in highly regulated environments and government. Strong, phishing-resistant, but requires PKI infrastructure.

**Temporary Access Pass (TAP)** 🎫: A time-limited passcode generated by an admin and given to a user. Used for bootstrapping authentication when a user has no registered methods yet, or for recovery scenarios. Not a primary method. Should expire quickly.

## 🚫 Methods to Disable

**SMS for high-value accounts**: If you have to allow SMS (for users without smartphones or reliable internet), limit it to lower-sensitivity accounts. Disable it for accounts with access to administrative functions.

**TOTP without phishing-resistant alternatives**: Not a reason to disable TOTP entirely, but a reason to prioritize deploying phishing-resistant alternatives for admin accounts.

**Security questions**: Legacy SSPR-only method. Knowledge-based authentication has well-documented weaknesses (answers are guessable, findable on social media, shareable). Disable security questions for SSPR and replace with any other method.

## 🔒 Method Registration Management

The authentication methods report in Entra ID shows which users have which methods registered. This is the data that tells you:

- What percentage of users have a phishing-resistant method registered (useful for planning enforcement)
- Which users have only SMS or voice registered (candidates for upgrading)
- Which users have no MFA registered at all (immediate priority)
- Which users have only a single method registered (vulnerable if they lose that method)

Two registered methods is the minimum for resilience. A user who loses their only phone and has no backup method is locked out until an admin intervenes.

## 🎯 The Method Migration Path

Organizations typically move through these phases:

1. Any MFA (SMS, voice, TOTP, Authenticator push) for all users
2. Microsoft Authenticator as primary, with number matching enforced
3. Passwordless via Authenticator for standard users
4. FIDO2 or Windows Hello for privileged accounts
5. Phase out SMS and voice for all accounts

Each phase improves the security posture. Not every organization needs to reach phase 5 for every user. The right target depends on the account's access level and the sensitivity of the resources it can reach.
