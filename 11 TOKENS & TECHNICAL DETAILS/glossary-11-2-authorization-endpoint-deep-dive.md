# Authorization Endpoint (Deep Dive)
*What's Actually in That Sign-In URL*

**Part of Entra ID Glossary Series: Glossary#11.2 - Authorization Endpoint (Deep Dive)**

---

A developer pasted a sign-in URL into a browser to test authentication manually. The URL was 300 characters long. They recognized `client_id` and `redirect_uri`. Everything else looked like noise.

Each parameter in that URL has a specific purpose. Some are required. Some are optional but critical for security. Some tell Entra ID exactly what kind of authentication experience to provide. Understanding every parameter is understanding how to control the authentication flow.

## 🌐 The Endpoint Itself

Entra ID's authorization endpoint follows the format:

```
https://login.microsoftonline.com/{tenant}/oauth2/v2.0/authorize
```

The `{tenant}` segment determines the authentication context:

**A specific tenant ID or domain** 🔵: `https://login.microsoftonline.com/contoso.com/oauth2/v2.0/authorize` limits authentication to accounts in that specific tenant. Users from other organizations can't sign in.

**`common`** 🌐: Accepts work, school, and Microsoft personal accounts from any tenant. Used for multi-tenant applications. Entra ID figures out which tenant to authenticate against based on the user's email.

**`organizations`** 🏢: Work and school accounts only, any tenant. Personal accounts excluded.

**`consumers`** 👤: Microsoft personal accounts only (Outlook.com, Hotmail, Live.com).

The endpoint choice is the first access control decision in the flow.

## 📋 Required Parameters

**`client_id`**: The application ID of the registered application. Tells Entra ID which application is initiating the flow. Entra ID uses this to look up the app registration: valid redirect URIs, requested permissions, token configuration.

**`response_type`**: `code` for the authorization code flow. This tells Entra ID to issue an authorization code, not tokens directly.

**`redirect_uri`**: Where Entra ID sends the user after authentication. Must exactly match a URI registered in the app registration. Case-sensitive. Trailing slash matters. Query strings generally not allowed. A mismatch here is one of the most common integration failures.

**`scope`**: The permissions being requested. Space-separated list of scope values. `openid` is required for OIDC flows (returns an ID token). `profile` adds name and picture claims. `email` adds the user's email. `offline_access` requests a refresh token. Application-specific scopes like `api://myapp/.default` or `User.Read` grant access to specific APIs.

## 🔐 Security Parameters

**`state`**: A random value generated by the application, included in the request, and returned by Entra ID in the response. The application validates that the returned state matches what it sent. This prevents CSRF attacks where an attacker's authorization code is injected into a victim's session. MSAL handles this automatically; custom implementations must implement it.

**`nonce`**: A random value included in the authorization request and embedded in the resulting ID token. The application validates the nonce claim in the token against the value it sent. Prevents token replay attacks. Required when requesting an ID token.

**`code_challenge` and `code_challenge_method`**: PKCE parameters. The code challenge is derived from the `code_verifier` (a random string generated locally). `code_challenge_method` should be `S256` (SHA-256 hash). These parameters make the authorization code only exchangeable by the client that generated the challenge.

## 🎛️ Experience Parameters

**`prompt`**: Controls the authentication experience.
- `none`: No interactive UI. Returns an error if the user isn't already silently authenticated. Used for silent token acquisition.
- `login`: Forces a fresh authentication even if the user has an active session. Useful for step-up scenarios.
- `consent`: Forces the consent screen even if the user has already consented.
- `select_account`: Shows an account picker even if there's one active session.

**`login_hint`**: Pre-fills the username field with the provided email address. Removes the account selection step when the application already knows who's signing in.

**`domain_hint`**: Skips the home realm discovery step and directs authentication to a specific IdP. `domain_hint=organizations` skips the personal account options. `domain_hint=contoso.com` directs federated users directly to their IdP.

**`max_age`**: Requires that the authentication was performed within the specified number of seconds. If the existing session is older, Entra ID forces re-authentication. Useful for high-sensitivity operations that require fresh credentials regardless of session age.

## ⚠️ What Happens with Errors

When the authorization request fails, Entra ID redirects back to the redirect URI with error parameters instead of a code. Common errors: `access_denied` (user cancelled or consent wasn't granted), `invalid_request` (malformed parameters), `unauthorized_client` (the application isn't allowed to use this flow).

These errors come back in the URL, not as HTTP error codes. The application must check for error parameters in the redirect response, not just look for the `code` parameter.
