# Passwordless Auth (Advanced)
*Deployment Reality Beyond the Demo*

📚 **Part of Entra ID Glossary Series: Glossary#13.5 - Passwordless Auth (Advanced)**

---

The demo always looks smooth. Admin enables passwordless in the policy. User opens the Authenticator app. Taps the number match. Signs in. No password. Applause.

Then the deployment starts. Shared workstations where Windows Hello enrollment doesn't make sense. Older applications that can't handle modern authentication. Users who lose their phone and are suddenly locked out of everything. FIDO2 keys that the organization bought but nobody can figure out how to provision. The "last mile" applications that still prompt for a password regardless of how Entra ID authenticated the user.

Passwordless deployment at scale is a program, not a feature toggle.

## 🔑 The Passwordless Methods in Entra ID

There are three primary passwordless authentication methods with meaningfully different deployment profiles:

**Microsoft Authenticator (phone sign-in)** 📱: Number matching push notification. User sees a number on their computer, taps the matching number in the Authenticator app, and is signed in. Requires a smartphone and the Authenticator app. Works well for office workers. Doesn't work for users without smartphones or in environments where phones are prohibited.

**FIDO2 security keys** 🔐: Hardware tokens (YubiKey, Feitian, etc.) that store private keys in hardware. Phishing-resistant: the key cryptographically verifies the site before signing. Works in shared workstation environments where Windows Hello enrollment per user is impractical. Requires physical key management: provisioning, distribution, loss replacement, reset procedures.

**Windows Hello for Business** 💻: Biometric (face, fingerprint) or PIN authentication that replaces the password for Windows sign-in and Entra ID authentication. Device-bound: the credential is stored on the specific device. Excellent user experience for dedicated device environments. Doesn't work for shared workstations or non-Windows devices.

**Passkeys** 🔑: The evolving standard that brings FIDO2 to consumer contexts. Passkeys can be device-bound (like FIDO2 keys) or synced across devices via platform keychain (iCloud Keychain, Google Password Manager). Microsoft Authenticator now supports passkeys, enabling phishing-resistant authentication from mobile without a separate hardware key.

## 🚀 Temporary Access Pass: The Enrollment Bootstrap

The most overlooked piece of passwordless deployment is the bootstrap problem. How does a user enroll in Windows Hello or register a FIDO2 key when they're new and have no authentication method yet?

Temporary Access Pass (TAP) is the answer. A TAP is a time-limited, one-time-use passcode generated by an administrator or via automated workflow. The new user authenticates with the TAP to get into My Security Info and register their passwordless method. The TAP expires (typically after 1-8 hours, configurable) and can never be used again.

TAP makes passwordless registration possible without the chicken-and-egg problem: the user gets a TAP in their welcome packet, uses it to register Windows Hello or a FIDO2 key, and from that point forward has no need for a password.

TAP is also the recovery mechanism when a user loses their phone or their FIDO2 key. Without TAP, the recovery path is a temporary password reset (which re-establishes a password credential). With TAP, the recovery is passwordless all the way.

## 📊 Authentication Strength Enforcement

Deploying passwordless methods is half the work. The other half is ensuring users actually use them.

Authentication strength policies in Conditional Access define which authentication methods satisfy a requirement. An authentication strength requiring "Phishing-resistant MFA" accepts FIDO2 keys, Windows Hello for Business, and Certificate-Based Authentication, but rejects push notifications and SMS (which are susceptible to phishing).

This is the enforcement mechanism for passwordless adoption. Policies can target specific high-risk applications with a phishing-resistant authentication strength requirement, driving adoption of FIDO2 or Windows Hello for those applications while allowing phone sign-in for lower-risk applications during the transition period.

## ⚠️ The Last-Mile Problem

Not every application that users access supports modern authentication. Legacy applications that accept only form-based username/password, applications that use NTLM or Kerberos for on-premises access, and some thick-client applications built before modern auth was standard all create exceptions to the passwordless story.

For these applications, the practical solutions are:

**Application Proxy with SSO**: Expose on-premises applications through Application Proxy, which authenticates users with Entra ID (passwordless) and then handles the legacy authentication to the back-end application on their behalf.

**Azure AD Domain Services for Kerberos/NTLM**: Applications that need Kerberos tickets can get them from Domain Services, which accepts modern Entra ID tokens and issues Kerberos tickets.

**Honest exception tracking**: Document which applications can't support passwordless and manage those exceptions explicitly. Not every application is convertible on the same timeline.
